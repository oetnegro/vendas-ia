# lib/email Overview

Integração direta com Gmail API. Hoje é mono-provider; vai virar adapter pattern junto com `lib/google` no épico 2.1 (Outlook/Microsoft 365) — abstração `lib/email/provider.ts` planejada.

## Arquivos

- **`gmail.ts`** — núcleo da integração Gmail. Funções públicas:
  - `sendGmailMessage(...)` — envia via `users/me/messages/send`. Renderiza HTML, monta headers RFC822, retorna `id`/`threadId`.
  - `getGmailThread(...)` — leitura de thread completa (usada no sync).
  - `searchGmailMessages(...)` — busca por query (ex: `from:{lead.email} newer_than:30d`) quando o Gmail agrupou resposta em outra thread.
  - `refreshGoogleAccessToken(refreshToken)` — troca refresh token por access token novo.
  - `cleanEmailReply(value)` — remove histórico citado (`Em ... escreveu`, linhas `>`, mensagem original) pra Inbox ficar legível.
  - `extractEmailAddress(value)` / `isAutomaticEmailMessage(...)` — utilitários de parsing/filtragem.

- **`gmail-sync.ts`** — `syncWorkspaceGmail(...)` orquestra leitura das threads conhecidas, dedupe por `(workspace_id, gmail_message_id)`, criação/atualização de `email_threads`/`email_messages` e detecção inbound vs outbound comparando `From` com o e-mail Google conectado.

## Fluxos principais

**Envio outbound:**
1. Engine ou rota chama `refreshGoogleAccessToken` se o token expirou.
2. `sendGmailMessage` monta RFC822 + headers + body HTML (vindo de `lib/template.ts → renderRichTextHtml`).
3. Resposta Gmail traz `id` e `threadId` → caller grava em `email_messages` e cria/atualiza `email_threads`.

**Sync inbound:**
1. `syncWorkspaceGmail` itera threads do workspace que já têm `gmail_thread_id`.
2. `getGmailThread` busca mensagens; se não encontra resposta no mesmo thread, `searchGmailMessages` busca por `from:{lead.email}`.
3. Mensagens novas → upsert em `email_messages` por unique index `(workspace_id, gmail_message_id)`.
4. Atualiza `email_threads.last_message_at`, muda lead para `replied` quando inbound.
5. Dispara `lib/ai/email-reply-engine` quando há inbound novo + `email_threads.ai_enabled=true`.

## Cuidados

- **Nunca** chamar essas funções no client. Sempre em rota server-side com `lib/supabase-admin.ts`.
- Tokens vivem em `google_connections` (será `email_connections` após 2.1). Os campos têm sufixo `_ciphertext` mas hoje **não estão realmente criptografados** — épico 3.2 aplica AES-GCM real.
- Toda operação de envio precisa registrar em `activity_logs` (variáveis usadas, assunto, IDs Gmail).
- Trava anti-duplicidade: nunca enviar a mesma `(lead_id, cadence_step_id)` se já existe `send_jobs` `queued`/`running`/`sent`.
